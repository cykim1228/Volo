# volo-spec — Volo 제품 명세 작성

planner 에이전트가 Volo의 PRD를 작성·갱신할 때 따르는 방법론.

## 컨텍스트 확인 (먼저)
1. `docs/PRD.md`, `_workspace/01_planner_prd.md` 존재 여부 확인.
   - 없으면 **초기 작성**. 있으면 읽고 **부분 갱신**(변경 요청·신규 정보 반영분만).
2. 사용자 확정 방향을 입력으로 받는다: 독립 앱 + SRT 내보내기 / 로컬 faster-whisper / 부가가치 4종(한국어 교정·줄바꿈·타이밍 최적화·번역·스타일 프리셋).

## 작성 원칙
- **고통 → 기능 매핑.** 각 기능 옆에 "없애는 수작업"을 한 줄로 적는다. 매핑 안 되는 기능은 비목표.
- **MVP는 핵심 가치 경로 하나.** 영상 투입 → faster-whisper 전사 → 타임코드 SRT 출력 → 프리미어 임포트. 이것이 동작하면 자막 노동의 큰 덩어리가 사라진다. 부가가치는 이 경로 위에 단계적으로 얹는다.
- **수용 기준은 측정 가능.** "정확하다" 같은 모호한 말 금지. "10분 한국어 영상에서 cue가 시간순으로 정렬되고 겹침이 없으며, 각 cue가 SRT 표준(인덱스/타임코드/텍스트/빈 줄)을 만족한다" 처럼 검증 가능하게.

## PRD 필수 구조
1. **문제·타깃** — 누가, 어떤 편집 상황에서, 무엇이 고통인가. 프리미어 기본 자동자막 대비 Volo의 차별점(한국어 정확도·교정·줄바꿈/타이밍 최적화·번역·스타일).
2. **핵심 사용자 플로우** — 단계별(투입→옵션→생성→미리보기/편집→내보내기→임포트). 각 단계의 입력/결과.
3. **기능 목록 (우선순위 표)** — 컬럼: 기능 | 설명 | 없애는 수작업 | 단계(MVP/P2/P3) | 우선순위(P0~P2).
4. **기능별 요구사항** — 동작 상세, 옵션, 경계 조건.
5. **기능별 수용 기준** — 기능마다 검증 가능한 완료 조건. QA가 이 항목을 그대로 검증한다.
6. **비목표** — 이번에 하지 않는 것(예: 실시간 라이브 자막, 클라우드 협업).
7. **용어 정의** — SRT, VTT, CPS(characters per second), 글로서리, cue, 세그멘테이션, 번인(burn-in) 등.

## 권장 기능 단계화 (시작점, 조정 가능)
- **MVP (P0)**: 오디오 추출(ffmpeg) → faster-whisper 전사(단어 타임스탬프) → CPS/줄길이 기반 세그멘테이션 → SRT 내보내기 → CLI 동작.
- **Phase 2**: 한국어 교정 + 글로서리, 자막 미리보기/간단 편집 데스크톱 UI, VTT/프리미어 캡션 내보내기, 스타일 프리셋.
- **Phase 3**: 번역/다국어 동시 출력, 배치 처리, 화자 분리(선택).
